Allocation คืออะไร?

หลักการพื้นฐานของระบบกระเป๋าเงิน Nexum — อ่านแล้วทุกคนในทีมควรเข้าใจตรงกัน

📄 อ่านประมาณ 5 นาที 👥 สำหรับทุกคนในทีม 🔄 อัปเดต ต.ค. 2568
1
แนวคิดหลัก
เปรียบเหมือน
กระเป๋าสตางค์ที่มีช่องแยกอยู่ภายใน

ลองนึกถึงกระเป๋าสตางค์ที่มีเงิน 1,000 บาท แล้วคุณแบ่งไว้ในใจว่า "300 บาทนี้ไว้จ่ายค่าน้ำมัน" — เงินยังอยู่ในกระเป๋าเดิม ไม่ได้หยิบออกไปไหน แค่คุณ "จอง" มันไว้แล้ว

Allocation ทำงานแบบเดียวกัน — มันคือการ "จอง" เงินในกระเป๋าหลักไว้สำหรับวัตถุประสงค์เฉพาะ โดยที่เงินจริงๆ ยังอยู่ที่เดิม

หลักการสำคัญที่สุด
ยอดเงินในกระเป๋าหลักไม่เปลี่ยน เมื่อสร้าง Allocation — แค่แบ่งว่าเงินส่วนนั้นใช้ทำอะไรได้บ้าง และเหลือเท่าไหร่ที่ใช้ได้ทันที
ตัวอย่าง: กระเป๋าหลักมี 10,000 บาท
แบ่งการใช้งาน รวม 10,000 THB
Payroll 5,000
Pool 1,000
ใช้ได้ 4,000
Payroll alloc — จองไว้จ่ายเงินเดือน
Lending pool — จองไว้ปล่อยกู้
ใช้ได้ทันที — free balance
สูตรง่ายๆ
ยอดเงินทั้งหมด = เงินที่จองไว้ (Allocation) + เงินที่ใช้ได้ทันที (Free balance)
10,000 = 6,000 (จอง) + 4,000 (ใช้ได้)
2
ควบคุมพฤติกรรมด้วย 2 สวิตช์

แต่ละ Allocation มีสวิตช์ 2 ตัวที่เปิด/ปิดได้อิสระ ไม่เกี่ยวกัน กำหนดว่ากระเป๋านี้จะ "ทำงานอย่างไร"

Revolving
วงเงินจะ "เติมคืน" หรือเปล่า?
เปิด

ใช้แล้วถ้าได้เงินคืน วงเงินกลับมาอัตโนมัติ เช่น บัตรเครดิต จ่ายคืนแล้วใช้ใหม่ได้

ปิด

ใช้แล้วหมดเลย วงเงินไม่กลับคืน เหมือนกองทุนที่ให้ครั้งเดียว

Restricted
เงินในกระเป๋านี้ "ถอนได้" หรือเปล่า?
เปิด

ล็อกไว้ ถอนออกหรือเอาไปใช้อื่นไม่ได้ ต้องใช้ตามวัตถุประสงค์เท่านั้น

ปิด

ยืดหยุ่น ยังถอนออกหรือใช้ที่อื่นได้ เป็นแค่การ "แนะนำ" ไม่ได้บังคับ

Revolving + Restricted

3
กระเป๋าอยู่ได้นานแค่ไหน?

เมื่อสร้าง Allocation เราต้องกำหนดว่ามันจะ "หมดอายุ" หรือเปล่า

⏳ Ongoing
ใช้ไปเรื่อยๆ จนกว่าจะยกเลิกเอง ไม่มีวันหมดอายุ เหมาะกับ Payroll, Credit pool
📅 Time-bound
กำหนดวันหมดอายุ พอครบกำหนด เงินที่เหลือจะ "unlock" กลับมาที่กระเป๋าหลัก
4
วงเงินเติมยังไง? (Revolving เท่านั้น)

กระเป๋าที่เปิด Revolving จะเติมวงเงินได้ 2 แบบ

Repayment-based
วงเงินเติมเมื่อผู้ใช้ชำระเงินคืน
→ ใช้กับระบบปล่อยกู้
ชำระ 100 บาท กลับมา 100 บาท
Schedule-based
วงเงินเติมตามวันที่กำหนด
→ ใช้กับระบบจ่ายเงินเดือน
ทุกวันที่ 25 เติม 5,000 บาทอัตโนมัติ
กรณีเงินไม่พอเติม (Schedule)
ถ้าถึงวันที่กำหนดแล้วเงินในกระเป๋าหลักไม่พอ → ไม่เติม และเปลี่ยน status เป็น pending_refill พร้อมแจ้งเตือนเจ้าของ รอจนเติมเงินพอแล้วค่อยดำเนินการ

3 กรณีที่ใช้จริงในระบบ
Case 1 — ระบบจ่ายเงินเดือน
บริษัทล็อกงบ แล้วเติมให้พนักงานทุกเดือนอัตโนมัติ
วิธีทำงาน
1
บริษัทสร้าง Payroll alloc จอง 5,000 บาทในกระเป๋าหลัก
2
ทุกวันที่ 25 ระบบดึงเงินจากกระเป๋าหลัก เติมเข้า alloc อัตโนมัติ
3
ถ้าเงินไม่พอ → แจ้งเตือนบริษัทให้เติม แล้วค่อยดำเนินการ
4
พนักงานใช้จ่ายผ่าน alloc นี้ บริษัทเห็นยอดใช้จริง real-time
✓ Revolving (เติมทุกเดือน) ✓ Restricted (ถอนไม่ได้) Schedule · วันที่ 25
Case 2 — ระบบปล่อยกู้
Issuer สร้าง pool แล้วดึงออกให้ลูกค้าทีละคน
วิธีทำงาน (2 ชั้น)
1
Issuer สร้าง Lending pool alloc จอง 10,000 บาทไว้ปล่อยกู้
2
เมื่อจะปล่อยกู้ให้ User A → ดึงเงินออกจาก pool สร้าง Credit alloc ให้ User A
3
User A ใช้จ่ายผ่าน Credit alloc — pool ลดลง
4
User A ชำระเงินคืน → Credit alloc เติม + pool เติมกลับอัตโนมัติ
✓ Revolving (เติมเมื่อ repay) ✓ Restricted (ถอนไม่ได้) Repayment-based

กฎที่ระบบบังคับเสมอ
เงินพอก่อนสร้าง Allocation
ก่อนสร้าง Allocation ใหม่ ระบบจะตรวจว่า free balance เพียงพอ ถ้าไม่พอ สร้างไม่ได้
free_balance ≥ allocation_amount
ยอดรวม Allocation ห้ามเกินยอดกระเป๋าหลัก
ผลรวมของทุก Allocation ต้องไม่เกินยอดเงินทั้งหมดในกระเป๋าหลัก
sum(allocations) ≤ main_wallet.balance
Restricted = ห้ามถอนออกเด็ดขาด
กระเป๋าที่เปิด Restricted ไว้ เงินส่วนนั้นถอนออกไม่ได้ ใช้ได้แค่ตามวัตถุประสงค์ที่กำหนด
เติมแบบ All-or-nothing
ถ้าเงินไม่พอเติม ระบบจะไม่เติมบางส่วน — เติมทั้งหมดหรือไม่เติมเลย และแจ้งเตือนเจ้าของ

ความหมาย

available_amount
วงเงินที่เหลือใช้ได้ใน allocation นั้น (= amount − used)
free_balance

Revolving

Restricted
เงินที่จองไว้ถอนออกไม่ได้ ต้องใช้ตามวัตถุประสงค์เท่านั้น
pending_refill
สถานะรอเติม — ถึงเวลาแล้วแต่เงินไม่พอ รอเจ้าของเติมเงิน
